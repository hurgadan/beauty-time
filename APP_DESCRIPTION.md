# Beauty-Time: описание приложения и его назначения

## 1. Что это за продукт

`Beauty-Time` — это SaaS-сервис для салонов красоты и частных мастеров, который решает задачу онлайн-записи клиентов и ведения рабочего расписания в одном месте.

Продукт состоит из двух основных интерфейсов:
- CRM для бизнеса (салон/мастер)
- Booking-интерфейс для клиента по ссылке

Главная идея: клиент записывается сам 24/7, а бизнес получает структурированное расписание, напоминания и подтверждение визитов, что снижает пропуски и потери выручки.

## 2. Назначение продукта

Назначение `Beauty-Time`:
- упростить процесс записи для клиента
- убрать ручную рутину для мастера/администратора
- снизить `no-show` через напоминания и подтверждение
- хранить клиентскую историю и базовую CRM-информацию
- дать возможность работать как салону с несколькими мастерами, так и мастеру-одиночке

## 3. Для кого продукт

Основная аудитория:
- частные мастера (парикмахер, мастер маникюра, педикюра, массажист и т.д.)
- небольшие и средние салоны красоты

Ключевой рынок запуска:
- Европа
- приоритет: Германия

## 4. Какие проблемы решает

До внедрения:
- запись ведется в мессенджерах, заметках, таблицах или вручную
- много времени тратится на переписки и звонки
- клиенты забывают о визите
- хаос в расписании и риск накладок

После внедрения:
- клиент сам выбирает услугу, мастера и время
- запись автоматически попадает в календарь CRM
- клиент получает напоминания и подтверждает визит
- мастер видит единое расписание и историю клиента

## 5. Ключевая ценность

Для бизнеса:
- меньше пропущенных визитов
- выше загрузка расписания
- меньше ручной операционной работы
- прозрачность по записям и клиентам

Для клиента:
- быстрый и понятный процесс записи
- возможность выбрать удобное время
- напоминания перед визитом
- простое подтверждение без сложной регистрации

## 6. Основные функциональные блоки (v1)

### CRM (для салона/мастера)
- управление услугами (цена, длительность, буферы)
- управление мастерами и расписанием
- календарь записей
- создание/редактирование/отмена записи
- карточка клиента и история посещений
- настройка email-напоминаний

### Booking (для клиента)
- переход по ссылке на запись
- выбор услуги
- выбор мастера или вариант "любой доступный"
- выбор даты и времени
- ввод контактов
- подтверждение через magic link + OTP
- экран успешной записи

### Напоминания и подтверждение
- email после создания записи
- email-напоминания `T-24h` и `T-2h`
- подтверждение визита клиентом из напоминания

## 7. Формат работы для разных бизнесов

### Мастер-одиночка
- может полностью работать с телефона
- ведет расписание, клиентов и услуги без компьютера

### Салон с командой
- несколько мастеров в одном workspace
- распределение по услугам и графикам
- единый календарь и клиентская база

## 8. Что не входит в текущую версию

В v1 не входит:
- Whitepages-платформа
- SMS/WhatsApp/Telegram уведомления
- продвинутая аналитика
- сложные маркетинговые инструменты

Эти функции планируются на следующих этапах развития.

## 9. Принципы продукта

- Mobile-first для клиентской записи
- Полноценная мобильная CRM (не урезанная относительно desktop)
- Простота ключевых действий
- Безопасная работа с персональными данными
- Локализация под рынок Германии (DE-first)

## 10. Безопасность и данные

- клиентская идентификация через magic link + OTP
- ограниченное хранение персональных данных (только необходимое)
- разграничение доступа по ролям в CRM
- соответствие GDPR-подходу для европейского рынка

## 10.1. Организация БД-слоя в backend

- Сущности хранятся внутри модулей: `src/modules/<module>/dao/*.entity.ts`.
- Миграции хранятся рядом с модулем: `src/modules/<module>/dao/migrations/*.migration.ts`.
- Запуск миграций выполняется через `database/data-source.js`.
- В миграциях используем TypeORM API (`Table`, `TableIndex`, `TableForeignKey`), без raw SQL.
- В миграциях держим только минимально необходимые индексы (в первую очередь уникальные бизнес-ограничения).
- В entity-классах сохраняем `@Index` как наглядное описание модельных индексов.

## 10.2. Организация доступа к данным в backend

- Используется слоистая схема: `controller -> service -> repository`.
- `service` слой не работает с TypeORM напрямую и не содержит SQL/запросов к БД.
- Для каждого модуля есть отдельные `*.repository.ts` провайдеры, где инжектятся TypeORM `Repository<Entity>`.
- Репозиторий модуля не должен работать с сущностями других модулей напрямую.
- Если модулю A нужна логика/данные модуля B, это делается через `service` модуля B (через импорт модуля и экспорт провайдера), а не через `InjectRepository` чужой `Entity`.
- Пример: в `auth` используется отдельный `OtpSessionRepository` только для `otp_sessions`, а данные tenant/client получаются через `TenantService` и `ClientsService`.
- Пример: в `booking` tenant/service/staff/client резолвятся через соответствующие сервисы модулей, а booking-репозитории отвечают за операции с appointment-потоком.
- По умолчанию используются типобезопасные методы TypeORM Repository (`find`, `findOneBy`, `save`, `delete`, `countBy` и т.д.).
- `QueryBuilder` используется только как исключение для сложных запросов, где Repository API становится избыточно громоздким.
- Отдельного технического модуля `CRM` нет: API разнесено по доменным модулям (`appointments`, `services`, `staff`, `clients`), а CRM-маршруты публикуются под префиксом `/crm/*`.
- В `repository` размещается только логика доступа к данным.
- В `service` размещаются бизнес-правила и проверки доступа (в том числе tenant ownership для update/delete).
- `service` возвращает сущности или данные доменного уровня, без DTO-маппинга.

## 10.3. DTO и API-контракты

- Контракты хранятся в `api/src/contracts`.
- Контракты разделены на 2 пакетных entrypoint'а:
  - `api/src/contracts/public/index.ts` (`@hurgadan/beauty-time-public-contracts`);
  - `api/src/contracts/crm/index.ts` (`@hurgadan/beauty-time-crm-contracts`).
- Типы организованы по модулям: `src/contracts/types/<module>`.
- Каждый тип хранится в отдельном файле `*.type.ts`.
- API-клиенты разделены по пакетам:
  - `src/contracts/public/clients` (public API);
  - `src/contracts/crm/clients` (CRM API).
- DTO контроллеров лежат в модульных папках `src/modules/<module>/dto`.
- DTO имплементируют соответствующие интерфейсы из `@contracts`.
- Request DTO валидируются через `class-validator`.
- Для обязательных полей request DTO используется non-null assertion (`!`).
- Маппинг сущностей в response DTO выполняется в контроллерах через `transformToDto`.
- В response DTO, которые проходят через `transformToDto`, поля помечаются `@Expose()`.
- `transformToDto` работает в строгом режиме `excludeExtraneousValues: true`.
- Списковые API в CRM возвращают массивы (`T[]`), без обертки `{ items: [...] }`.

## 10.4. Разделение публичных и CRM контроллеров

- Публичные точки API реализованы отдельными `public-*` контроллерами в доменных модулях:
  - booking endpoints (`/book/*`) в `booking/public-booking.controller.ts`;
  - client auth endpoints (`/auth/client/*`) в `auth/public-auth.controller.ts`.
- CRM endpoints остаются в доменных модулях и работают под префиксом `/crm/*`.
- Staff login (`/auth/staff/login`) относится к CRM и реализован отдельным CRM-контроллером.

## 11. Критерии успеха MVP

- клиент записывается меньше чем за 2 минуты
- заметное снижение пропусков визитов
- мастер экономит время на коммуникации и координации
- расписание ведется централизованно и без накладок

## 12. Результат для бизнеса

`Beauty-Time` превращает запись на услуги из ручного процесса в управляемую систему:
- больше подтвержденных визитов
- меньше организационных ошибок
- выше операционная эффективность
- готовая база для дальнейшего роста (аналитика, дополнительные каналы, расширения)
